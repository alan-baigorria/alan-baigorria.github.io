# Guía rápida: crt.sh

## ¿Qué es?

`crt.sh` es un buscador de certificados SSL/TLS registrados en los logs públicos de **Certificate Transparency**.

Se utiliza para descubrir:

- Subdominios.
- Dominios antiguos.
- Certificados vencidos.
- Nombres expuestos accidentalmente.

Es una técnica de reconocimiento pasivo porque consulta información pública sin escanear directamente el objetivo.

## ¿Cómo funciona?
 
Cada vez que una CA emite un certificado TLS, lo registra en logs públicos de Certificate Transparency. Ese certificado puede cubrir varios dominios a la vez a través del campo **SAN (Subject Alternative Name)**:
 
```text
example.com
www.example.com
api.example.com
admin.example.com
```
 
`crt.sh` indexa todos esos nombres. Al buscar un dominio, devolvés todos los certificados históricos que lo mencionan, incluyendo subdominios que quizás no aparecen por DNS ni en Google.
 
Lo importante: aparece **todo lo que alguna vez tuvo un certificado**, incluso si el subdominio ya no existe o el certificado venció. Por eso es una mina de oro para mapear superficie de ataque.
 
## Uso por interfaz web
 
Entrás a `https://crt.sh` y buscás. El comodín `%` funciona como en SQL:
 
- `%.example.com` devuelve todos los subdominios de `example.com`.
- `%.example.%` amplía a cualquier TLD (`.com`, `.com.ar`, `.net`, etc.).
- `example` a secas busca coincidencias parciales en cualquier dominio (ruidoso, pero útil para descubrir dominios relacionados).
URL directa para el wildcard más común:
 
```text
https://crt.sh/?q=%25.example.com
```
 
(`%25` es el `%` codificado para la URL).
 
## Uso por JSON (lo interesante para automatizar)
 
crt.sh expone la salida en JSON agregando `&output=json`. Combinado con `jq` sacás una lista limpia de subdominios únicos:
 
```bash
curl -s --retry 5 --retry-delay 5 --max-time 180 \
  'https://crt.sh/?q=%25.nasa.gov&output=json' \
  | jq -r '.[].name_value' | sed 's/\*\.//g' | sort -u > subdominios.txt
```

## Limitaciones
 
- **Muestra historia, no estado actual.** Un subdominio puede figurar y estar caído hace años. Siempre validá con resolución DNS y HTTP.
- **No cubre certificados internos.** Si la organización usa una CA privada que no publica en CT, esos nombres no aparecen.
- **No detecta subdominios sin certificado.** Los que van solo por HTTP plano o nunca tuvieron TLS no están.
- **Puede tener lag.** Un certificado recién emitido puede tardar en indexarse.
- **Timeouts en dominios grandes.** Para objetivos con miles de certificados, usá la salida JSON o el acceso SQL en vez de la web.
## Tips para bug bounty y pentesting
 
- Buscá también variantes del nombre de la organización, no solo el dominio principal: `%.empresa-staging.com`, `%.empresa-dev.com`.
- Los subdominios de `dev`, `test`, `staging`, `uat`, `admin`, `vpn`, `jenkins`, `git` suelen ser los más jugosos y peor protegidos.
- Cruzá los hallazgos con Wayback Machine y con fuentes de DNS pasivo para reconstruir superficie que ya no está en CT.
- Guardá snapshots periódicos: comparar la salida de crt.sh en el tiempo te avisa cuando la org levanta infraestructura nueva.
- Respetá siempre el scope del programa. Que un subdominio aparezca en crt.sh no significa que esté dentro del alcance autorizado.
## Referencia rápida de queries
 
| Búsqueda | Query |
|---|---|
| Todos los subdominios | `%.example.com` |
| Cualquier TLD | `%.example.%` |
| Coincidencia parcial | `example` |
| Un subdominio puntual | `api.example.com` |
| Salida JSON | agregar `&output=json` a la URL |
